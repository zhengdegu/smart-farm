## 一、网关定位与职责

```
传感器/执行器 ←──本地通信──→ 边缘网关 ←──MQTT/TLS──→ 云端(EMQX+后端)
                              │
                    ┌─────────┼─────────┐
                    │         │         │
               数据缓存   本地规则   设备管理
               (断网保障)  (紧急响应)  (协议转换)
```

### 1.1 核心职责

| 职责 | 说明 |
|------|------|
| 协议转换 | Modbus/RS485/LoRa → MQTT 统一上报 |
| 数据缓存 | 断网时本地缓存传感器数据，恢复后批量上传 |
| 本地规则引擎 | 紧急响应不依赖云端（如湿度过低立即开阀） |
| 设备管理 | 本地设备注册、心跳检测、状态维护 |
| 安全兜底 | 最大灌溉时长硬限制、异常自动关阀 |

## 二、选型方案

### 2.1 推荐方案：工业网关 + 自研轻量固件

| 项目 | 选型 | 理由 |
|------|------|------|
| 硬件 | 有人科技 USR-G807 / 映翰通 InGateway502 | 工业级、支持4G+WiFi+以太网、RS485/Modbus接口、-40~70℃工作温度 |
| 操作系统 | Linux (OpenWrt/Debian) | 可部署自研程序 |
| 运行时 | Java (GraalVM native-image) 或 Python | 与后端技术栈统一 |
| 本地存储 | SQLite | 轻量、可靠、断电不丢数据 |
| MQTT客户端 | Eclipse Paho | 成熟稳定 |

### 2.2 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| EMQX Edge | 与云端EMQX无缝对接 | 资源占用较大(需512MB+RAM) | 大型基地 |
| 自研网关程序 | 完全可控、轻量 | 开发维护成本 | MVP推荐 |
| EdgeX Foundry | 功能全面 | 过于复杂、资源占用大 | 不推荐 |
| 树莓派+自研 | 灵活便宜 | 非工业级、稳定性差 | 仅实验室 |

## 三、本地规则引擎

### 3.1 规则类型

```yaml
# 网关本地规则配置文件 rules.yaml
rules:
  # 紧急响应规则（断网时仍执行）
  - id: emergency_close_on_flood
    trigger: "sensor.soil_moisture > 95"
    action: "valve.close"
    priority: EMERGENCY
    description: "土壤湿度过高，紧急关阀防涝"

  - id: emergency_open_on_drought
    trigger: "sensor.soil_moisture < 20"
    action: "valve.open"
    params:
      duration_min: 15
    priority: EMERGENCY
    description: "土壤极度干旱，紧急灌溉15分钟"

  # 安全兜底规则（始终生效）
  - id: max_irrigation_guard
    trigger: "valve.open_duration_min > 60"
    action: "valve.force_close"
    priority: CRITICAL
    description: "灌溉超过60分钟强制关阀"

  # 定时规则（云端下发，本地执行）
  - id: scheduled_irrigation
    trigger: "cron(0 6 * * *)"  # 每天6:00
    action: "valve.open"
    params:
      duration_min: 20
    priority: NORMAL
    cloud_managed: true
```

### 3.2 规则执行引擎

```java
// 网关本地规则引擎（轻量实现）
public class LocalRuleEngine {
    private List<Rule> rules;
    
    public void evaluate(SensorData data) {
        for (Rule rule : rules) {
            if (rule.matches(data)) {
                // 检查是否与当前执行中的指令冲突
                if (rule.getPriority().higherThan(currentCommandPriority())) {
                    execute(rule.getAction());
                    reportToCloud(rule, data); // 网络恢复后上报
                }
            }
        }
    }
}
```

## 四、断网数据同步协议

### 4.1 数据缓存策略

```
断网时：
1. 传感器数据写入本地 SQLite（带时间戳）
2. 控制指令执行记录写入本地
3. 本地规则触发记录写入本地
4. 缓存容量上限：1GB（约可存储7天数据）
5. 超出容量时，按FIFO淘汰最旧的传感器数据（保留控制记录）

恢复后：
1. 检测到网络恢复（MQTT连接成功）
2. 上报网关状态：离线时长、缓存数据量、本地执行记录
3. 批量上传缓存数据（按时间顺序，每批100条，间隔100ms）
4. 云端确认接收后，清除本地缓存
5. 同步云端最新规则配置到本地
```

### 4.2 时间同步

```
问题：断网期间网关时钟可能漂移

方案：
1. 网关内置 RTC（实时时钟）芯片，断电不丢时间
2. 网络恢复后立即 NTP 同步
3. 上报数据携带网关本地时间戳 + 同步偏移量
4. 云端根据偏移量修正历史数据时间
```

## 五、网关运维

### 5.1 远程管理

| 功能 | 实现方式 |
|------|---------|
| 固件OTA升级 | MQTT下发升级指令 + HTTP下载固件包 + 校验 + 回滚 |
| 配置下发 | MQTT Topic: `/{tenant}/gateway/{gw_id}/config` |
| 远程重启 | MQTT指令，带延迟执行（防止误操作） |
| 日志上传 | 按需触发，上传最近N小时日志到OSS |
| 健康检查 | 每5分钟上报：CPU/内存/磁盘/网络/温度 |

### 5.2 固件升级流程

```
1. 云端推送升级通知（版本号 + 下载URL + SHA256）
2. 网关下载固件包到临时分区
3. 校验 SHA256
4. 写入备份分区（A/B分区方案）
5. 重启切换到新分区
6. 新固件启动后30秒内上报成功 → 标记为活跃分区
7. 如果30秒内未上报 → 自动回滚到旧分区
```
