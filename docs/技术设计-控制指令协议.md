## 一、指令状态机

```
                    ┌─────────────────────────────────┐
                    │                                 ↓
[CREATED] → [SENT] → [DELIVERED] → [EXECUTING] → [COMPLETED]
               │          │              │
               ↓          ↓              ↓
           [TIMEOUT]  [TIMEOUT]     [FAILED]
               │          │              │
               ↓          ↓              ↓
           [RETRY 1] [RETRY 2]     [ESCALATED]
               │          │
               ↓          ↓
           [RETRY 2] [RETRY 3]
               │          │
               ↓          ↓
           [RETRY 3] [ABANDONED] → 告警通知人工介入
               │
               ↓
           [ABANDONED]
```

### 1.1 状态定义

| 状态 | 说明 | 超时时间 |
|------|------|---------|
| CREATED | 指令已创建，等待发送 | — |
| SENT | 已发送到 EMQX | 30秒 |
| DELIVERED | 设备已收到（MQTT QoS1 PUBACK） | 60秒 |
| EXECUTING | 设备正在执行（设备主动上报） | 根据动作类型 |
| COMPLETED | 执行完成，设备确认 | — |
| TIMEOUT | 超时未收到确认 | — |
| FAILED | 设备报告执行失败 | — |
| ABANDONED | 重试耗尽，放弃执行 | — |
| ESCALATED | 已升级为人工处理 | — |

### 1.2 超时与重试策略

```java
@Configuration
public class CommandRetryConfig {
    // 最大重试次数
    public static final int MAX_RETRY = 3;
    // 各阶段超时时间

        Duration.ofSeconds(30),  // 第2次重试
        Duration.ofSeconds(60)   // 第3次重试
    };
}
```

## 三、硬件安全兜底

### 3.1 最大灌溉时长限制

```
安全机制（三层防护）：

Layer 1 - 软件层（后端）
  └─ 定时任务每分钟检查执行中的指令
  └─ 超过 max_duration_min 自动发送 CLOSE_VALVE

Layer 2 - 网关层（本地）
  └─ 网关本地维护灌溉计时器
  └─ 超过 60 分钟无论如何关阀（硬编码上限）
  └─ 断网时此机制仍然生效

Layer 3 - 硬件层（设备端）
  └─ 电磁阀n  └─ 断电自动关阀（常闭型电磁阀）
  └─ 最终安全兜底，不依赖任何软件
```

### 3.2 异常流量检测

```java
// 灌溉过程中持续监测流量
@Scheduled(fixedRate = 60000) // 每分钟检查
public void checkIrrigationAnomaly() {
    List<Command> executing = commandRepo.findByStatus("EXECUTING");
    for (Command cmd : executing) {
        SensorData flow = sensorService.getLatest(cmd.getDeviceId(), "flow");
        // 流量异常高 → 可能管道破裂
        if (flow.getValue() > cmd.getExpectedFlow() * 3) {
            commandService.emergencyStop(cmd);
            alertService.send(AlertLevel.CRITICAL, "流量异常，已紧急关阀");
        }
        // 流量为0但阀门开启 → 可能阀门故障
        if (getValue() == 0 && cmd.getElapsedMinutes() > 2) {
            alertService.send(AlertLevel.WARNING, "阀门已开启但无流量，请检查");
        }
    }
}
```

## 四、冲突仲裁

### 4.1 指令冲突规则

| 场景 | 仲裁规则 |
|------|---------|
| 同一阀门同时收到开和关 | 后到的指令排队，等前一个完成 |
| 自动触发 vs 手动操作 | 手动操作优先级高，覆盖自动指令 |
| 紧急停止 vs 任何指令 | 紧急停止最高优先级，立即执行 |
| 多人同时操作同一设备 | 先到先得 + 锁定机制（执行中不接受新指令） |

### 4.2 指令队列

```java
// 设备级指令队列，保证串行执行
@Service
public class CommandQueueService {
    private final Map<String, Queue<Command>> deviceQueues = new ConcurrentHashMap<>();
    
    public synchronized boolean enqueue(Command cn        String deviceId = cmd.getDeviceId();
        // 紧急停止直接插队
        if (cmd.getPriority() == Priority.EMERGENCY) {
            cancelAllPending(deviceId);
            return executeImmediate(cmd);
        }
        // 检查是否有执行中的指令
        if (hasExecutingCommand(deviceId)) {
            deviceQueues.computeIfAbsent(deviceId, k -> new LinkedList<>()).add(cmd);
            return false; // 排队中
        }
        return executeImmediate(cmd);
    }
}
```

## 五、指令数据模型

```sql
CREATE TABLE command_log (
    id BIGSERIAL PRIMARY KEY,
    cmd_id VARCHAR(64) UNIQUE NOT NULL,
    tenant_id BIGINT NOT NULL,
    device_id VARCHAR(64) NOT NULL,
    action VARCHAR( NOT NULL,
    params JSONB,
    constraints JSONB,
    status VARCHAR(16) NOT NULL DEFAULT 'CREATED',
    priority INTEGER DEFAULT 0,
    retry_count INTEGER DEFAULT 0,
    -- 时间追踪
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    sent_at TIMESTAMPTZ,
    delivered_at TIMESTAMPTZ,
    executing_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ,
    -- 结果
    result JSONB,
    error_message TEXT,
    -- 操作人
    triggered_by VARCHAR(32), -- 'AUTO'/'USER:xxx'
    INDEX idx_device_status (device_id, status),
    INDEX idx_created_at (created_at)
);
```
