# 物联网智慧农业平台 — 产品需求文档（PRD）

> 版本：V4.0 | 日期：2026-04-23 | 基于CEO评审V2：补充三端功能规划 + 数据报表 + 作物模板 + 路线图调整

## 一、项目定位

### 1.1 一句话定位

面向设施温室蔬菜种植基地，提供"设备接入 + 智能灌溉 + 数据驱动决策 + 三端协同（小程序 / PC 管理后台 / 3D 数据大屏）"的一体化智慧灌溉解决方案。

### 1.2 聚焦场景：设施温室蔬菜

**为什么选温室蔬菜：**

- 单位面积产值高（年产值 8-15 万/亩），客户付费意愿强
- 灌溉频次高、水肥敏感，节水节肥效果可量化
- 环境相对可控，传感器部署标准化程度高
- 全国设施蔬菜面积超 3700 万亩，市场基数大

**典型客户画像：**

| 维度 | 描述 |
|------|------|
| 规模 | 50-500 亩设施温室 |
| 作物 | 番茄、黄瓜、辣椒、叶菜类 |
| 痛点 | 灌溉靠经验、用水浪费 30%+、人工巡棚成本高、异常发现滞后 |
| 决策人 | 基地负责人 / 合作社理事长 / 农业园区管理者 |
| 技术水平 | 会用微信和智能手机，不熟悉复杂软件 |

### 1.3 行业痛点（聚焦灌溉场景）

| 痛点 | 现状 | 量化影响 |
|------|------|---------|
| 灌溉凭经验 | 浇多浇少全靠感觉 | 水资源浪费 30-40% |
| 异常响应慢 | 人工巡棚发现问题平均滞后 2-4 小时 | 作物损失不可逆 |
| 远程管理难 | 必须到现场操作阀门 | 人工成本高，夜间无人值守 |
| 数据缺失 | 无历史数据积累 | 无法优化种植策略 |

## 二、市场规模

### 2.1 TAM / SAM / SOM

```
TAM（总可触达市场）
  全国设施农业智慧化市场 ≈ 800 亿元/年
  （设施农业面积 4000 万亩 × 平均 2000 元/亩智慧化投入）

SAM（可服务市场）
  设施温室蔬菜智慧灌溉市场 ≈ 120 亿元/年
  （设施蔬菜 3700 万亩 × 15% 规模化基地 × 2000 元/亩）

SOM（短期可获取市场）
  目标区域（山东、河北、云南、广东）≈ 3-5 亿元/年
  （4 省设施蔬菜规模化基地约 15-25 万亩 × 2000 元/亩）
```

### 2.2 市场趋势

- 2025 年中央一号文件明确"加快智慧农业发展"，各省配套补贴政策密集出台
- 设施农业水肥一体化渗透率不足 20%，增量空间大
- 农业劳动力持续减少，自动化刚需增强

## 三、竞争分析与壁垒

### 3.1 竞品对比

| 竞品 | 模式 | 价格带 | 短板 |
|------|------|--------|------|
| 托普云农 | 硬件为主，软件弱 | 5-20 万/套 | 软件体验差，无智能决策 |
| 大气候 | 大田场景为主 | SaaS 月费制 | 温室场景覆盖弱 |
| 本地集成商 | 项目制，一次性交付 | 10-50 万/项目 | 无持续服务，系统废弃率高 |

### 3.2 竞争壁垒：垂直场景深度服务 + 数据飞轮

**短期壁垒（0-12 个月）**：垂直场景深度服务能力。我们不做通用 IoT 平台，而是深扎温室灌溉场景，提供从传感器选型、灌溉策略到客户成功的全链条服务。巨头做广度，我们做深度。

**长期壁垒（12 个月+）**：数据飞轮 + 客户成功网络。

```
客户使用 → 积累灌溉数据 → 优化灌溉模型 → 节水效果更好
    ↑                                          ↓
    └──── 客户续费 + 口碑转介绍 ←── 可量化的ROI ←┘
```

**数据飞轮**：每个客户的灌溉数据都在训练我们的模型。服务 10 个温室后，第 11 个客户开箱即用的灌溉策略就比竞品更精准。数据越多，模型越准，客户越难迁移。

**客户成功案例**：农业客户决策高度依赖"隔壁老王用了效果怎么样"。每个成功案例都是下一个客户的最强销售工具。我们的目标是让每个客户都成为可量化的标杆案例（节水 X%、省人工 Y 个）。

## 四、商业模式

### 4.1 收入结构

| 收入项 | 定价 | 说明 |
|--------|------|------|
| 轻量体验包（一次性） | 0.8-1 万元 | 含 2-3 个传感器 + 1 个控制阀 + 小程序基础功能，适合小规模试用验证 |
| 硬件套件（一次性） | 3-5 万元/套 | 含传感器、控制阀、网关、安装调试 |
| 年度服务费 | 5,000-8,000 元/年 | 含平台使用、数据存储、微信告警、远程支持 |
| 政府示范项目 | 15-30 万元/项目 | 含定制开发、培训、验收报告 |

### 4.2 单位经济模型

```
单客户生命周期价值（LTV）：
  硬件毛利：1-2 万（毛利率 30-40%）
  服务费 3 年：1.5-2.4 万（毛利率 80%+）
  LTV ≈ 2.5-4.4 万

客户获取成本（CAC）：
  BD 拜访 + 示范演示 + 安装调试 ≈ 0.8-1.5 万
  
LTV/CAC ≈ 2.5-3x（健康）
```

### 4.3 生存策略

**前 6 个月靠政府示范项目养活团队**：

- 目标拿下 2-3 个区县级智慧农业示范项目（每个 15-30 万）
- 示范项目同时作为产品打磨场景和标杆案例
- 服务费收入从第 7 个月开始贡献现金流

## 五、客户获取路径与漏斗

### 5.1 获客漏斗

```
认知（Awareness）
  ├─ 政府示范项目展示 → 周边基地参观学习
  ├─ 农业展会/现场观摩会
  ├─ 农技站/农资经销商转介绍
  └─ 短视频平台（抖音/快手农业号，展示实际节水效果和智能灌溉场景）
        ↓
兴趣（Interest）
  ├─ 现场演示（带客户看运行中的示范基地）
  └─ 提供 2 周免费试用（装 2-3 个传感器体验）
        ↓
决策（Decision）
  ├─ 出具 ROI 测算报告（预计节水 X 吨、省人工 Y 个）
  └─ 提供同区域成功案例数据
        ↓
购买（Purchase）
  └─ 硬件套件 + 年度服务合同
        ↓
续费 & 转介绍（Retention & Referral）
  ├─ 季度效果报告（实际节水数据 vs 预期）
  └─ 老客户转介绍奖励（减免服务费）
```

### 5.2 关键转化率目标

| 漏斗阶段 | 目标转化率 | 关键动作 |
|----------|-----------|---------|
| 认知 → 兴趣 | 30% | 示范基地参观 |
| 兴趣 → 试用 | 50% | 免费装 2-3 个传感器 |
| 试用 → 付费 | 25-30% | ROI 报告 + 案例背书 |
| 付费 → 续费 | 80%+ | 季度效果报告 |
| 付费 → 转介绍 | 30% | 老带新减免政策 |

## 六、MVP 功能范围（扩展版）

MVP 做 8 件事，覆盖"数据采集 → 智能决策 → 远程控制 → 效果量化"完整闭环，三端协同交付：

### 6.1 设备接入与数据采集

| 项目 | 说明 |
|------|------|
| 支持传感器 | 土壤湿度、土壤温度、空气温湿度（4 种够用） |
| 接入协议 | MQTT（覆盖 90% 农业传感器） |
| 数据频率 | 每 5 分钟采集一次 |
| 数据展示 | 手机端实时传感器当前值和 24 小时趋势 |

### 6.2 智能灌溉自动执行

| 项目 | 说明 |
|------|------|
| 规则引擎 | 基于土壤湿度阈值自动触发灌溉（如湿度 < 40% 开阀，> 70% 关阀） |
| 策略模板 | 预置番茄、黄瓜、叶菜 3 套灌溉策略，开箱可用 |
| 定时任务 | 支持设置每日定时灌溉计划 |
| 执行记录 | 每次灌溉的时间、时长、触发原因自动记录 |

### 6.3 手机远程控制（微信小程序）

| 项目 | 说明 |
|------|------|
| 入口 | 微信小程序（无需下载 APP，降低使用门槛） |
| 核心操作 | 手动开/关灌溉阀、查看传感器数据、查看灌溉记录 |
| 紧急操作 | 一键停止所有灌溉 |
| 多用户 | 支持基地负责人 + 2-3 个操作员账号 |

### 6.4 微信告警通知

| 项目 | 说明 |
|------|------|
| 告警类型 | 传感器离线、湿度异常、灌溉执行失败、设备故障 |
| 通知方式 | 微信服务号模板消息（实时推送） |
| 告警规则 | 可配置阈值和通知对象 |
| 免打扰 | 支持设置夜间免打扰时段（紧急告警除外） |

### 6.5 数据报表与效果量化（新增）

数据报表是客户续费的核心依据，也是政府项目验收的必要交付物。没有报表，节水效果无法量化，"数据飞轮"无从谈起。

| 项目 | 说明 |
|------|------|
| 灌溉统计 | 日/周/月灌溉次数、总时长、估算用水量（基于阀门流量 × 时长） |
| 节水率对比 | 系统灌溉 vs 历史人工灌溉用水量对比，自动计算节水百分比 |
| 传感器趋势 | 7 天 / 30 天土壤湿度、温度趋势折线图，支持多传感器叠加 |
| 告警统计 | 按类型、级别、时间段统计告警数量和处理时效 |
| 月度报告 | 自动生成月度运营报告（PDF），包含节水数据、灌溉记录、设备健康度 |
| 数据导出 | 支持 CSV / Excel 导出，满足政府项目报告需求 |

**数据模型扩展：**

```sql
-- 灌溉统计日汇总表（定时任务每日凌晨聚合）
CREATE TABLE irrigation_daily_summary (
    id BIGSERIAL PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    greenhouse_no VARCHAR(20),
    summary_date DATE NOT NULL,
    irrigation_count INTEGER DEFAULT 0,       -- 当日灌溉次数
    total_duration_min INTEGER DEFAULT 0,     -- 总灌溉时长（分钟）
    estimated_water_liters DOUBLE PRECISION,  -- 估算用水量（升）
    avg_soil_moisture DOUBLE PRECISION,       -- 当日平均土壤湿度
    auto_trigger_count INTEGER DEFAULT 0,     -- 自动触发次数
    manual_trigger_count INTEGER DEFAULT 0,   -- 手动触发次数
    created_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(tenant_id, greenhouse_no, summary_date)
);

-- 月度报告记录
CREATE TABLE monthly_report (
    id BIGSERIAL PRIMARY KEY,
    tenant_id BIGINT NOT NULL,
    report_month VARCHAR(7) NOT NULL,         -- 格式：2026-04

    total_irrigation_count INTEGER,
    total_water_liters DOUBLE PRECISION,
    device_online_rate DOUBLE PRECISION,      -- 设备在线率

    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 6.6 作物灌溉模板与生长周期管理（新增）

作物模板是"智能决策"的最低门槛。不同作物、不同生长阶段的灌溉需求差异巨大，硬编码阈值无法覆盖。作物模板让系统从"自动开关阀"升级为"懂农业的灌溉顾问"。

| 项目 | 说明 |
|------|------|
| 预置模板 | 番茄、黄瓜、叶菜（生菜/菠菜）3 套完整灌溉策略 |
| 生长周期 | 每套模板按生长阶段划分：苗期 → 生长期 → 开花期 → 果期 → 采收期 |
| 阶段参数 | 每个阶段独立配置：湿度上下限、灌溉时长、灌溉频次、最佳温度范围 |
| 自动切换 | 用户设定种植日期后，系统按天数自动切换生长阶段和对应灌溉参数 |
| 自定义模板 | 支持基于预置模板修改参数，保存为自定义模板 |
| 灌溉建议 | 基于当前生长阶段 + 实时传感器数据，给出"建议灌溉 / 暂不需要"提示 |

### 7.3 网关层设计

农业现场网络不稳定是常态，网关是系统可靠性的关键一环。

#### 7.3.1 网关选型建议

| 候选方案 | 优势 | 适用场景 |
|----------|------|---------|
| 树莓派 4B + EMQX Edge | 成本低（~500 元）、社区生态好、可运行 Java/Python | 示范项目、小规模部署（≤20 传感器） |
| 研华 WISE-710 | 工业级、宽温、4G/WiFi/以太网多网口 | 正式商用、恶劣环境 |
| 自研 ESP32 网关 | 极低成本（~100 元）、功耗低 | 大规模部署降本阶段 |

**MVP 阶段推荐**：树莓派 4B + EMQX Edge，兼顾成本与灵活性。商用阶段切换至研华工业网关。

#### 7.3.2 本地规则引擎（MVP：硬编码阈值触发）

网关内置轻量规则引擎，确保断网时灌溉策略仍可自主执行。

**MVP 阶段只做硬编码阈值触发**，不实现可视化规则编辑器或动态规则 DSL：

```
MVP 规则引擎能力（硬编码实现）：
├── 阈值触发：土壤湿度 < 下限 → 开阀，> 上限 → 关阀（阈值写死在配置文件中）
├── 定时执行：按预设时间表执行灌溉计划（cron 表达式硬编码）
├── 安全兜底：单次灌溉最大时长 60 分钟强制关阀（不可配置）
└── 规则同步：云端通过 MQTT 下发 JSON 配置文件更新阈值参数

MVP 不做：
├── ✗ 可视化规则编辑器（Phase 2）
├── ✗ 复杂条件组合（AND/OR 逻辑链）
├── ✗ 自定义规则 DSL
└── ✗ 机器学习自适应阈值
```

规则存储格式：JSON 文件，云端下发时带版本号，网关对比版本决定是否更新。阈值参数示例：

```json
{
  "version": "1.0.0",
  "rules": [
    { "sensor": "soil_moisture", "threshold_low": 40, "threshold_high": 70, "action": "valve_toggle" },
    { "type": "timer", "cron": "0 6,18 * * *", "duration_minutes": 20 }
  ],
  "safety": { "max_duration_minutes": 60 }
}
```

#### 7.3.3 断网数据同步协议

```
┌─────────────────────────────────────────────────────┐
│                 断网同步状态机                         │
├─────────────────────────────────────────────────────┤
│  在线模式：实时上报（QoS 1，每 5 分钟一条）            │
│      ↓ 网络断开检测（MQTT keepalive 超时 30s）        │
│  离线模式：                                          │
│    - 传感器数据写入本地 SQLite（按时间戳排序）          │
│    - 灌溉执行记录写入本地日志                         │
│    - 本地规则引擎接管灌溉决策                         │
│      ↓ 网络恢复检测（MQTT 重连成功）                  │
│  同步模式：                                          │
│    - 按时间戳顺序批量上报离线期间数据（每批 100 条）    │
│    - 服务端返回 ACK 后删除本地已同步数据               │
│    - 同步完成后切回在线模式                           │
└─────────────────────────────────────────────────────┘
```

本地存储容量：保留最近 7 天数据（约 2000 条/传感器），超出按 FIFO 淘汰。

### 7.5 MQTT Topic 命名规范

采用分层命名，便于 ACL 控制和消息路由：

```
Topic 结构：smartfarm/{tenant_id}/{gateway_id}/{device_type}/{device_id}/{action}

示例：
├── 传感器数据上报（设备 → 云端）
│   smartfarm/t001/gw01/sensor/soil_moisture_01/telemetry
│   smartfarm/t001/gw01/sensor/air_temp_01/telemetry
│
├── 控制指令下发（云端 → 设备）
│   smartfarm/t001/gw01/valve/valve_01/command
│   smartfarm/t001/gw01/valve/valve_01/command/ack
│
├── 设备状态上报（设备 → 云端）
│   smartfarm/t001/gw01/valve/valve_01/status
│   smartfarm/t001/gw01/sensor/soil_moisture_01/status
│
├── 网关心跳与管理
│   smartfarm/t001/gw01/$heartbeat
│   smartfarm/t001/gw01/$config          # 云端下发配置
│   smartfarm/t001/gw01/$config/ack      # 网关确认配置
│
└── 系统级 Topic
    $SYS/smartfarm/t001/gw01/connected   # LWT 遗嘱消息
    $SYS/smartfarm/t001/gw01/disconnected
```

**命名规则：**
- 全小写，单词间用下划线分隔
- `tenant_id` 为租户标识，预留多租户扩展
- `gateway_id` 为网关标识，一个网关管理多个设备
- `device_type`：sensor / valve / gateway
- `action`：telemetry（遥测）/ command（指令）/ status（状态）/ ack（确认）
- 系统管理 Topic 以 `$` 前缀标识

### 7.6 控制指令确认机制

灌溉控制指令涉及物理设备操作，必须有完整的确认和容错机制。

#### 7.6.1 指令状态机（5 种状态）

```
┌──────────┐   云端下发    ┌──────────┐   网关ACK    ┌──────────┐   执行完成   ┌──────────┐
│  已创建   │────────────→│  已发送   │───────────→│  已确认   │───────────→│  已执行   │
│ CREATED  │             │  SENT    │            │CONFIRMED │            │ EXECUTED │
└──────────┘             └──────────┘            └──────────┘            └──────────┘
                              │                       │
                              │ 30s超时重试≤3次         │ 执行异常/重试耗尽
                              │ 重试耗尽               ↓
                              └──────────────→┌──────────┐
                                              │  失败     │
                                              │ FAILED   │
                                              └──────────┘
                                                    │
                                                    ↓
                                              推送告警 + 人工介入
```

**5 种状态定义：**

| 状态 | 含义 | 触发条件 | 后续动作 |
|------|------|---------|---------|
| CREATED | 指令已创建，待下发 | 用户/规则引擎触发 | 入库记录，准备下发 |
| SENT | 指令已通过 MQTT 下发 | 云端 publish 成功 | 等待网关 ACK（超时 30s） |
| CONFIRMED | 网关已确认收到 | 网关返回 ACK | 等待执行结果（超时 120s） |
| EXECUTED | 设备执行成功 | 设备返回执行成功 | 记录完成，更新灌溉日志 |
| FAILED | 执行失败（终态） | 设备返回失败 / 重试耗尽 | 推送告警 + 人工介入 |

**超时处理：** SENT 状态 30s 无 ACK 自动重试（最多 3 次，指数退避），重试耗尽转 FAILED。CONFIRMED 状态 120s 无执行结果同理。

#### 7.6.2 超时重试策略

```
重试策略：
├── 最大重试次数：3 次
├── 重试间隔：指数退避 — 30s / 60s / 120s
├── 重试条件：TIMEOUT 状态（非 FAILED）
├── 重试幂等：每条指令携带唯一 command_id，设备端去重
└── 重试耗尽：标记 FAILED，推送微信告警给负责人
```

#### 7.6.3 硬件安全兜底

即使软件层出现异常，硬件层必须有独立的安全保护：

| 安全措施 | 实现方式 | 说明 |
|----------|---------|------|
| 最大灌溉时长 | 网关本地定时器，60 分钟强制关阀 | 防止指令丢失导致持续灌溉 |
| 阀门看门狗 | 硬件级 watchdog，90 分钟无心跳自动关阀 | 网关宕机时的最后防线 |
| 水流量上限 | 流量计检测，超过阈值自动关阀 | 防止管道破裂导致水灾 |
| 手动急停 | 现场物理按钮，一键断电关阀 | 任何情况下的人工干预手段 |

## 八、安全设计

### 8.1 设备认证（TLS + Token）

```
设备接入认证流程：
┌─────────┐         TLS 1.2+          ┌──────────┐
│  网关    │◄─────────────────────────►│  EMQX    │
│         │  双向证书验证（mTLS）       │  Broker  │
└─────────┘                           └──────────┘
     │                                      │
     │  MQTT CONNECT (username=device_id,   │
     │  password=device_token)              │
     └──────────────────────────────────────┘
```

**认证方案：**
- 传输层：TLS 1.2+ 加密，网关预置 CA 证书，生产环境启用 mTLS 双向认证
- 应用层：设备连接时携带 device_id + device_token（出厂预烧录，支持云端轮换）
- Token 管理：每台设备独立 Token，支持远程吊销和定期轮换（90 天）
- 防重放：MQTT Client ID 唯一，EMQX 拒绝重复 Client ID 连接

### 8.2 MQTT ACL 权限控制

基于 EMQX 内置 ACL 机制，实现最小权限原则：

| 客户端类型 | 允许发布（Publish） | 允许订阅（Subscribe） |
|-----------|--------------------|--------------------|
| 传感器网关 | `smartfarm/{自身tenant}/{自身gw}/sensor/+/telemetry`<br>`smartfarm/{自身tenant}/{自身gw}/+/+/status` | `smartfarm/{自身tenant}/{自身gw}/+/+/command`<br>`smartfarm/{自身tenant}/{自身gw}/$config` |
| 后端服务 | `smartfarm/+/+/+/+/command` | `smartfarm/#`（全量订阅） |
| 小程序（通过后端代理） | 不直接连接 MQTT | 不直接连接 MQTT |

**ACL 规则：**
- 设备只能发布/订阅自身 tenant 和 gateway 下的 Topic
- 跨租户隔离：ACL 严格限制 tenant_id 匹配
- 后端服务使用独立超级用户账号，权限独立管理
- 小程序不直接连接 MQTT，通过后端 REST API 间接操作

### 8.3 API 鉴权（JWT）

后端 REST API 采用 JWT 无状态鉴权：

```
鉴权流程：
1. 用户登录 → 后端验证账号密码 → 签发 JWT（有效期 2 小时）
2. 请求携带 Authorization: Bearer <token>
3. 后端验证签名 + 过期时间 + 权限声明

JWT Payload 结构：
{
  "sub": "user_id",
  "tenant": "tenant_id",
  "role": "admin|operator",
  "permissions": ["device:read", "valve:control", "config:write"],
  "exp": 1714000000,
  "iat": 1713992800
}
```

**安全措施：**
- 签名算法：RS256（非对称，公钥可分发给网关验证）
- Access Token 有效期：2 小时
- Refresh Token 有效期：7 天，存储于 HttpOnly Cookie
- 权限粒度：基于 RBAC，区分管理员/操作员角色
- 敏感操作（阀门控制）需二次确认或短信验证码

### 8.4 控制指令签名

所有下发至设备的控制指令必须携带数字签名，防止伪造：

```json
{
  "command_id": "cmd_20260420_001",
  "action": "valve_open",
  "target": "valve_01",
  "params": { "duration_minutes": 30 },
  "timestamp": 1713600000,
  "nonce": "a3f8b2c1",
  "signature": "HMAC-SHA256(command_id+action+target+timestamp+nonce, device_secret)"
}
```

**签名规则：**
- 算法：HMAC-SHA256，密钥为设备独立 secret（出厂预置）
- 防重放：timestamp 有效窗口 ±5 分钟 + nonce 去重（Redis 缓存 10 分钟）
- 网关验签失败：拒绝执行，上报安全告警

### 8.5 数据加密

| 层级 | 方案 | 说明 |
|------|------|------|
| 传输加密 | TLS 1.2+（MQTT）/ HTTPS（API） | 所有通信链路加密 |
| 存储加密 | PostgreSQL 透明数据加密（TDE） | 数据库文件级加密 |
| 敏感字段 | AES-256-GCM 应用层加密 | 设备 Token、用户密码等敏感字段 |
| 备份加密 | GPG 加密备份文件 | 防止备份泄露 |
| 密钥管理 | 环境变量 + HashiCorp Vault（Phase 2） | MVP 阶段用环境变量，后续引入 Vault |

### 9.2 日志方案

```
日志架构：
├── 应用日志：Logback → JSON 格式 → 本地文件（按天滚动，保留 30 天）
├── 访问日志：Nginx access log → JSON 格式
├── MQTT 日志：EMQX 内置日志 → 本地文件
├── 审计日志：关键操作（阀门控制、配置变更）→ 独立审计表（不可删除）
└── 日志聚合（Phase 2）：Loki + Promtail → Grafana 统一查询
```

**日志规范：**
- 格式：JSON 结构化日志，包含 timestamp / level / trace_id / tenant_id / message
- 级别：ERROR（需立即处理）、WARN（需关注）、INFO（关键业务流程）、DEBUG（开发调试）
- 脱敏：日志中不记录设备 Token、用户密码等敏感信息
- trace_id：每个请求/指令全链路唯一标识，便于问题追踪

### 9.3 备份策略

| 备份对象 | 策略 | 保留周期 | 存储位置 |
|----------|------|---------|---------|
| PostgreSQL 全量 | 每日凌晨 2:00 pg_dump | 保留 30 天 | 异地对象存储（OSS） |
| PostgreSQL 增量 | WAL 归档，每小时同步 | 保留 7 天 | 异地对象存储 |
| EMQX 配置 | 配置变更时自动备份 | 保留 10 个版本 | Git 仓库 |
| 应用配置 | Git 版本管理 | 永久 | Git 仓库 |
| 网关本地数据 | 云端同步成功后保留 7 天 | 7 天 | 网关本地 SD 卡 |

**恢复目标：**
- RPO（恢复点目标）：≤ 1 小时（WAL 归档间隔）
- RTO（恢复时间目标）：≤ 2 小时（含数据库恢复 + 服务重启）

### 9.4 部署方案（Docker Compose）

MVP 阶段采用 Docker Compose 单机部署，简单可靠：

```yaml
# docker-compose.yml 核心服务编排
services:
  app:
    image: smartfarm/backend:latest
    ports: ["8080:8080"]
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - DB_URL=jdbc:postgresql://postgres:5432/smartfarm
    depends_on: [postgres, emqx, redis]
    restart: always
    deploy:
      resources:
        limits: { cpus: '2', memory: 2G }

  postgres:
    image: timescale/timescaledb:latest-pg15

      - POSTGRES_DB=smartfarm
    restart: always

  emqx:
    image: emqx/emqx:5.x
    ports: ["1883:1883", "8883:8883", "18083:18083"]
    volumes: ["emqx_data:/opt/emqx/data"]
    restart: always

  redis:
    image: redis:7-alpine
    volumes: ["redis_data:/data"]
    restart: always

  prometheus:
    image: prom/prometheus:latest
    volumes: ["./prometheus.yml:/etc/prometheus/prometheus.yml"]
    restart: always

  grafana:
    image: grafana/grafana:latest
    ports: ["3000:3000"]
    volumes: ["grafana_data:/var/lib/grafana"]

    image: nginx:alpine

volumes:
  pgdata:
  emqx_data:
  redis_data:
  grafana_data:
```

**部署规范：**
- 所有服务配置 `restart: always`，异常退出自动重启

## 十三、安全设计

> 详细方案见第八章，本节为顶层汇总。

### 13.1 设备认证（TLS + Token）

- 传输层：MQTT 全链路 TLS 1.2+ 加密，生产环境启用 mTLS 双向证书认证
- 应用层：每台设备独立 device_id + device_token，出厂预烧录，支持云端 90 天轮换与远程吊销
- 防重放：MQTT Client ID 唯一，EMQX 拒绝重复连接

### 13.2 MQTT ACL

- 最小权限原则：设备仅可发布/订阅自身 tenant + gateway 下的 Topic
- 跨租户严格隔离，后端使用独立超级用户账号
- 小程序不直连 MQTT，通过后端 REST API 代理

### 13.3 API 鉴权（JWT）

- RS256 非对称签名，Access Token 2 小时有效，Refresh Token 7 天（HttpOnly Cookie）
- RBAC 权限模型：管理员 / 操作员角色，敏感操作（阀门控制）需二次确认
- JWT Payload 包含 tenant_id + permissions 声明，后端逐请求校验

### 13.4 指令签名

- 所有控制指令携带 HMAC-SHA256 签名（密钥为设备独立 secret）
- 防重放：timestamp ±5 分钟有效窗口 + nonce 去重（Redis 缓存 10 分钟）
- 验签失败拒绝执行并上报安全告警

### 13.5 数据加密

| 层级 | 方案 |
|------|------|
| 传输 | TLS 1.2+（MQTT）/ HTTPS（API） |
| 存储 | PostgreSQL TDE 透明数据加密 |
| 敏感字段 | AES-256-GCM 应用层加密（设备 Token、用户密码） |
| 备份 | GPG 加密 |
| 密钥管理 | MVP 环境变量，Phase 2 引入 HashiCorp Vault |

## 十四、控制指令协议

> 详细方案见第七章第 6 节，本节为顶层汇总。

### 14.1 指令状态机（5 种状态）

```
CREATED ──(下发)──→ SENT ──(网关ACK)──→ CONFIRMED ──(执行完成)──→ EXECUTED
                     │                       │
                     │ 超时/重试耗尽           │ 执行异常/重试耗尽
                     └───────────────────────→ FAILED ──→ 告警 + 人工介入
```

五种状态：已创建（CREATED）→ 已发送（SENT）→ 已确认（CONFIRMED）→ 已执行（EXECUTED）/ 失败（FAILED）。

### 14.2 超时重试策略

- 最大重试：3 次，指数退避间隔 30s / 60s / 120s
- 仅 TIMEOUT 状态触发重试，FAILED 不重试
- 幂等保证：每条指令携带唯一 command_id，设备端去重
- 重试耗尽：标记 FAILED，推送微信告警

### 14.3 硬件安全兜底

| 措施 | 实现 |
|------|------|
| 最大灌溉时长 | 网关本地定时器，60 分钟强制关阀 |
| 阀门看门狗 | 硬件级 watchdog，90 分钟无心跳自动关阀 |
| 水流量上限 | 流量计超阈值自动关阀 |
| 手动急停 | 现场物理按钮一键断电 |

## 十五、网关层设计

> 详细方案见第七章第 3 节，本节为顶层汇总。

### 15.1 选型

| 阶段 | 方案 | 成本 | 适用场景 |
|------|------|------|---------|
| MVP / 示范 | 树莓派 4B + EMQX Edge | ~500 元 | ≤20 传感器，灵活开发 |
| 商用 | 研华 WISE-710 | 工业级定价 | 宽温、多网口、恶劣环境 |
| 规模降本 | 自研 ESP32 网关 | ~100 元 | 大规模部署 |

### 15.2 本地规则引擎（MVP：硬编码阈值）

网关内置轻量规则引擎，断网时灌溉策略仍可自主执行。**MVP 只做硬编码阈值触发，不做可视化规则编辑器或动态 DSL**：

- 阈值触发：土壤湿度 < 下限开阀，> 上限关阀（阈值硬编码在 JSON 配置文件中）
- 定时执行：按预设 cron 表达式灌溉（硬编码）
- 安全兜底：单次灌溉最大 60 分钟强制关阀（不可配置）
- 规则同步：云端通过 MQTT 下发 JSON 配置文件（带版本号），仅更新阈值参数

### 15.3 断网数据同步

- 在线模式：QoS 1 实时上报，每 5 分钟一条
- 离线检测：MQTT keepalive 超时 30s 进入离线模式
- 离线存储：传感器数据写入本地 SQLite（按时间戳排序），灌溉记录写入本地日志
- 恢复同步：网络恢复后按时间戳顺序批量上报（每批 100 条），服务端 ACK 后删除本地已同步数据
- 容量策略：保留最近 7 天数据（约 2000 条/传感器），超出 FIFO 淘汰

### 20.2 Testcontainers 方案

使用 Testcontainers 在 CI 中启动真实依赖，避免 Mock 与生产环境不一致：

```java
// 集成测试基类示例
@SpringBootTest
@Testcontainers
class IntegrationTestBase {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("timescale/timescaledb:latest-pg15");

    @Container
    static GenericContainer<?> emqx = new GenericContainer<>("emqx/emqx:5.x")
        .withExposedPorts(1883);

    @Container
    static GenericContainer<?> redis = new GenericContainer<>("redis:7-alpine")
        .withExposedPorts(6379);
}
```

**测试容器覆盖：**

| 容器 | 用途 | 说明 |
|------|------|------|
| TimescaleDB | 数据库集成测试 | 验证 SQL、时序查询、数据迁移 |
| EMQX | MQTT 消息测试 | 验证发布/订阅、QoS、ACL |
| Redis | 缓存/去重测试 | 验证 nonce 去重、Token 缓存 |

### 20.3 硬件联调测试

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| 传感器数据上报 | 真实传感器 → 网关 → 云端 | 数据完整入库，延迟 < 10s |
| 控制指令下发 | 云端 → 网关 → 阀门 | 指令执行成功率 ≥ 95% |
| 断网恢复 | 拔网线 → 本地缓存 → 恢复同步 | 数据零丢失 |

| 并发指令 | 同时下发 10 条指令 | 全部正确执行，无冲突 |

### 20.4 CI 流水线集成

```

                                                                    ↓
```

## 二十四、Git 分支策略与错误码规范

### 24.1 Git 分支策略

采用简化版 Git Flow，适合小团队快速迭代：

```
main（生产分支）
 │
 ├── develop（开发主线）
 │     │
 │     ├── feature/device-mqtt      # 功能分支
 │     ├── feature/irrigation-rule  # 功能分支
 │     └── feature/wechat-alert     # 功能分支
 │
 ├── hotfix/xxx                     # 紧急修复（从 main 拉出）
 └── release/v1.0.0                 # 发版分支（从 develop 拉出）
```

**分支规则：**

| 分支类型 | 命名规范 | 来源 | 合并目标 | 说明 |
|----------|---------|------|---------|------|
| main | main | — | — | 生产代码，保护分支，仅通过 PR 合并 |
| develop | develop | main | main（通过 release） | 开发主线，CI 必须通过 |
| feature | feature/{模块}-{功能} | develop | develop | 功能开发，完成后 PR 合并 |
| hotfix | hotfix/{问题描述} | main | main + develop | 紧急修复，双向合并 |
| release | release/v{版本号} | develop | main + develop | 发版准备，仅修 bug |

**Commit 规范（Conventional Commits）：**

```
feat: 新增传感器数据查询API
fix: 修复MQTT断连后重连失败
docs: 更新API接口文档
refactor: 重构指令状态机逻辑
test: 补充灌溉规则单元测试
chore: 升级Spring Boot版本
```

**PR 规则：**
- 至少 1 人 Code Review 通过
- CI 全部通过（单元测试 + 集成测试 + 代码检查）
- 禁止直接 push 到 main/develop

### 24.2 错误码规范

采用 6 位数字错误码，按模块分段：

```
错误码格式：XXYYYY
  XX   = 模块编号（01-99）
  YYYY = 模块内错误序号（0001-9999）
```

**模块分段：**

| 模块编号 | 模块名称 | 错误码范围 | 示例 |
|----------|---------|-----------|------|
| 10 | 通用/系统 | 100001-109999 | 100001 系统内部错误 |
| 11 | 认证/鉴权 | 110001-119999 | 110001 Token 过期 |
| 20 | 设备管理 | 200001-209999 | 200001 设备不存在 |
| 21 | 传感器数据 | 210001-219999 | 210001 数据格式异常 |
| 30 | 灌溉控制 | 300001-309999 | 300001 指令下发失败 |
| 31 | 规则引擎 | 310001-319999 | 310001 规则配置无效 |
| 40 | 告警通知 | 400001-409999 | 400001 微信推送失败 |
| 50 | 网关通信 | 500001-509999 | 500001 网关离线 |

**错误响应格式：**

```json
{
  "code": 300001,
  "message": "指令下发失败：设备离线",
  "detail": "device_id=valve_01, last_online=2026-04-20T10:30:00Z",
  "trace_id": "abc123def456"
}
```

**规范要求：**
- 所有 API 错误必须返回标准错误码，禁止裸抛异常信息
- 错误码在代码中以枚举类集中管理，禁止硬编码数字
- 错误消息面向开发者（中文），detail 字段提供排查线索
- trace_id 贯穿全链路，便于日志关联

### 26.4 技术实现要点

| 技术点 | 方案 |
|--------|------|

| 图表 | ECharts 5.x，深色主题，自适应大屏分辨率 |

## 二十九、AI 智能体功能规格（Phase 2-3）

### 29.1 战略定位

AI 智能体是智慧农业平台从"工具"升级为"顾问"的关键一步。竞品（托普云农、大气候）都停留在"看数据、按按钮"的阶段，没有一家做到"对话式交互 + 自主决策"。这是我们拉开差距的机会。

**核心理念：让不懂技术的农户，用说话的方式管理温室。**

```
竞品现状                    我们的目标
─────────                  ─────────
看数字 → 自己判断 → 手动操作    说一句话 → AI 判断 → 自动执行
                              ↓
                         "张叔，1号棚湿度偏低，
                          建议灌溉20分钟，要开吗？"
```

### 29.2 AI 能力分层（渐进式落地）

不一步到位，按数据积累和技术成熟度分三层推进：

| 层级 | 能力 | 技术方案 | 落地阶段 |
|------|------|---------|---------|
| L1 智能问答 | 自然语言查询数据、控制设备 | 大模型 + Function Calling | Phase 2（第 4-5 个月） |
| L2 主动巡检 | 定时巡检 + 异常主动通知 + 灌溉建议 | Agent 循环 + 规则引擎 + 大模型 | Phase 2（第 5-6 个月） |
| L3 自主决策 | 基于历史数据自动优化灌溉策略 | 统计模型 + 大模型推理 | Phase 3（第 8-10 个月） |

### 29.3 L1：对话式智能助手（Phase 2）

#### 29.3.1 产品形态

在微信小程序和 PC 后台中嵌入对话窗口，用户用自然语言与系统交互。

**小程序入口：** 首页底部浮动按钮"问问小农"，点击展开对话面板。
**PC 后台入口：** 右下角悬浮对话窗口，类似客服聊天。
**微信服务号：** 直接在服务号对话框中发消息，AI 回复（最自然的入口）。

#### 29.3.2 支持的对话场景

| 告警查询 | "今天有什么告警" | 调用告警 API → 按级别汇总 |

#### 29.3.5 安全约束

| 约束 | 说明 |
|------|------|
| 权限继承 | AI 助手继承当前用户的角色权限，操作员不能通过 AI 执行管理员操作 |
| 危险操作确认 | 阀门控制类操作必须二次确认："确定要打开2号棚阀门灌溉20分钟吗？回复'确认'执行" |
| 频率限制 | 每用户每分钟最多 10 次对话，防止滥用 |
| 审计日志 | 所有 AI 触发的操作记录到 audit_log，triggered_by 标记为 "AI:user_id" |
| 模型降级 | 大模型 API 不可用时，降级为关键词匹配的简单问答 |
| 输入过滤 | 过滤 prompt injection 攻击，限制输入长度 500 字符 |

#### 29.3.6 大模型选型

| 候选 | 优势 | 劣势 | 推荐场景 |
|------|------|------|---------|
| 通义千问（阿里） | 国内合规、中文好、Function Calling 支持好、有农业语料 | 英文弱 | **首选**，国内部署合规 |
| GLM-4（智谱） | 中文好、API 稳定、价格低 | Function Calling 不如通义 | 备选 |
| DeepSeek | 性价比极高、推理能力强 | 服务稳定性待验证 | 成本敏感场景 |
| GPT-4o（OpenAI） | 能力最强 | 合规风险、延迟高、成本高 | 仅用于内部测试对比 |

**推荐方案：** 通义千问作为主模型，DeepSeek 作为降级备选。通过抽象层封装，支持一键切换模型。

### 29.4 L2：AI 主动巡检 Agent（Phase 2 后期）

#### 29.4.1 产品形态

AI 不再只是被动回答问题，而是主动巡检温室状态，发现异常时通过微信主动通知农户。

**与现有告警的区别：** 现有告警是"阈值触发"（湿度 < 20% 才报警），AI 巡检是"趋势预判"（湿度在持续下降，按这个速度 2 小时后会低于阈值）。

#### 29.4.2 巡检任务

| 巡检项 | 频率 | AI 做什么 | 通知方式 |
|--------|------|----------|---------|
| 环境趋势预判 | 每 30 分钟 | 分析传感器趋势，预测未来 2-4 小时是否会触发告警 | 微信消息："1号棚湿度持续下降，预计2小时后需要灌溉" |
| 灌溉效果评估 | 每次灌溉后 1 小时 | 对比灌溉前后土壤湿度变化，评估灌溉是否有效 | 微信消息："上次灌溉效果良好，湿度从35%回升到62%" |
| 设备健康检查 | 每 2 小时 | 检查传感器数据是否异常（数据不变、跳变、离线） | 微信消息："3号棚土壤湿度传感器数据异常，可能需要检查" |
| 每日摘要 | 每日 20:00 | 汇总当日灌溉情况、用水量、异常事件 | 微信消息：今日摘要报告 |
| 生长阶段提醒 | 阶段切换时 | 提醒农户作物进入新阶段，灌溉参数已自动调整 | 微信消息："番茄进入开花坐果期，灌溉策略已自动调整" |

#### 29.4.4 System Prompt 设计

```
你是一位经验丰富的温室灌溉专家，负责监控和管理设施温室的灌溉系统。

你的职责：
1. 分析传感器数据趋势，预判未来 2-4 小时的环境变化
2. 评估灌溉效果，判断灌溉是否充分或过量
3. 检查设备健康状态，发现异常数据模式
4. 结合作物生长阶段，给出专业的灌溉建议

你的沟通风格：
- 用农户听得懂的语言，不用技术术语
- 直接说结论和建议，不要长篇大论
- 给出具体的数字和操作建议（"建议灌溉20分钟"而不是"适当灌溉"）
- 紧急情况用简短有力的语言

当前温室信息：
- 作物：{crop_type}，当前处于{current_stage}（第{day}天）
- 该阶段最佳土壤湿度范围：{moisture_low}%-{moisture_high}%
- 该阶段最佳温度范围：{temp_low}°C-{temp_high}°C
```

### 29.5 L3：自主决策引擎（Phase 3）

#### 29.5.1 能力描述

基于积累的历史灌溉数据，AI 自动优化灌溉策略参数。不需要复杂的机器学习模型，用统计方法 + 大模型推理就够。

| 能力 | 实现方式 | 说明 |
|------|---------|------|
| 阈值自动优化 | 统计分析 | 分析历史灌溉数据，找到"灌溉后湿度回升最快"的最佳触发阈值 |
| 灌溉时长优化 | 回归分析 | 根据"灌溉时长 vs 湿度回升量"的历史关系，推荐最优灌溉时长 |
| 季节性调整 | 时间序列 | 识别季节性模式，提前调整灌溉参数 |
| 异常模式识别 | 大模型推理 | 发现"灌溉后湿度不升反降"等异常模式，提示可能的管道泄漏或传感器故障 |

#### 29.5.2 数据要求

L3 需要至少 3 个月的连续灌溉数据才能开始训练。这就是为什么放在 Phase 3。

```
数据飞轮启动条件：
├── 最少 3 个月连续数据
├── 至少 1000 次灌溉记录
├── 至少 2 个完整生长周期
└── 传感器数据完整率 > 90%
```

| `/api/v1/ai/suggestions/{id}/reject` | POST | 拒绝优化建议 |

### 29.10 竞争壁垒分析

| 维度 | 竞品现状 | 我们的 AI 优势 |
|------|---------|---------------|
| 交互方式 | 点按钮、看数字 | 说话就能控制，零学习成本 |
| 异常发现 | 阈值告警（事后） | 趋势预判（事前） |
| 灌溉决策 | 固定阈值 | 基于数据 + 作物知识的动态建议 |
| 客户服务 | 人工客服 | AI 7×24 在线，秒级响应 |
| 数据价值 | 只存不用 | 数据驱动策略优化，越用越准 |
